================================================================================
 HISTÓRICO DO PROJETO EventOS AI
================================================================================
 Data: 16/07/2026
 Descrição: Plataforma completa de gestão de eventos (competidora DeskSoft/Facedoor)
 Arquitetura: Monorepo com 7 microserviços NestJS, Next.js, Python AI, Kafka, PostgreSQL, Redis
================================================================================

1. VISÃO GERAL DO QUE FOI CONSTRUÍDO
================================================================================

--- 1.1 BACKEND (7 microserviços NestJS) ---

  identity (porta 3001) - Autenticação e organização
    - Auth (register, login, refresh)
    - Users CRUD
    - Organizations CRUD + membros
    - JWT com refresh token
    - @eventos-ai/auth lib compartilhada

  event (porta 3002) - Gestão de eventos
    - CRUD eventos com publish/cancel/duplicate
    - Rooms CRUD
    - Schedules CRUD

  ticket (porta 3003) - Ingressos e pedidos
    - Ticket-types CRUD
    - Lots CRUD (Early Bird, Regular)
    - Coupons CRUD
    - Orders CRUD com confirm/refund

  checkin (porta 3004) - Credenciamento
    - Check-in por QR code e manual
    - Attendees CRUD + batch import
    - Face recognition (enroll + verify + liveness)
    - Sync endpoints
    - WebSocket gateways (checkin, notifications, chat)
    - Kafka producer (CHECKIN_CREATED)
    - Kafka consumer (ORDER_CONFIRMED, TICKET_ISSUED)

  crm (porta 3005) - CRM
    - Pipelines CRUD com stages
    - Deals CRUD com move/close/reopen
    - Contacts CRUD com merge/deduplication
    - Domain layer completa (entidades, repositórios, eventos de domínio)
    - 61 testes unitários
    - Kafka producer (DEAL_CREATED, DEAL_MOVED, DEAL_CLOSED, CONTACT_CREATED)
    - Kafka consumer (CHECKIN_CREATED, PAYMENT_COMPLETED)

  sponsor (porta 3006) - Patrocinadores [CRIADO]
    - Sponsors CRUD (tiers: diamond/gold/silver/bronze)
    - Booths CRUD com check-in tracking
    - Metrics com ROI calculation

  payment (porta 3007) - Pagamentos [CRIADO]
    - Payments CRUD com capture/refund
    - PIX generation + webhook
    - Boleto generation + webhook
    - Gateway Strategy pattern (Stripe, Asaas, Mercado Pago)
    - Kafka producer (PAYMENT_COMPLETED, PAYMENT_FAILED, PAYMENT_REFUNDED)
    - Kafka consumer (ORDER_CREATED)

--- 1.2 LIBS COMPARTILHADAS ---

  backend/libs/auth/ (@eventos-ai/auth)
    - AuthModule (forRoot dinâmico)
    - JwtAuthGuard
    - CurrentUser decorator
    - JwtPayload interface

  backend/libs/messaging/ (@eventos-ai/messaging) [CRIADO]
    - EventBusModule (forRoot, Global)
    - EventBusService (publish/subscribe via KafkaJS)
    - EventBusProducer (wraps KafkaJS Producer)
    - DomainEvent interface
    - Tópicos: CHECKIN, TICKET, CRM, PAYMENT, SPONSOR, EVENT, NOTIFICATION
    - Schemas: checkin.events, ticket.events, crm.events, payment.events

--- 1.3 API GATEWAY ---

  backend/gateways/api-gateway/ (porta 8080) [CRIADO]
    - Proxy dinâmico para todos os 7 serviços + AI
    - Cache Redis com invalidação
    - Rate limiting por endpoint (auth:5/min, checkin:60, ai:10, geral:100)
    - Response envelope padronizado: { success, data, meta }
    - Error handling global
    - Swagger em /v1/docs
    - Health check de todos os upstreams

4. ARQUITETURA DE EVENTOS (Kafka)
================================================================================

  Fluxo implementado:

  Ticket (ORDER_CONFIRMED) ──Kafka──→ Checkin (atualiza attendee)
  Checkin (CHECKIN_CREATED) ──Kafka──→ CRM (loga interação)
  Payment (PAYMENT_COMPLETED)──Kafka──→ CRM (loga atividade)
  Ticket (ORDER_CREATED) ──Kafka──→ Payment (cria payment auto)

  Cadeia completa:
    Order → Payment → Ticket → Check-in → CRM (activity log)
